dynamique:

les tables 'dynamiques' sont celles qui stocquent les champs oblig et facul ci-dessous
tables[0] doit contenir la PK qui est la Principal Primary Key du contexte
PK la Primary Key, PK[0] doit etre la PK de tables[0] et est la PPK du contexte. Elle sera retournée en valeur finale
FK la cle auxiliaire de tables[1], doit pointer(Foreign Key Pointe) sur PK[0] 

[dynamique] 
tables=spip_vap_humains,spip_vap_core 
PK=idhum,idvap 
FK=lienperso 
FKP=idhum

cles extra :

si la-les table(s) dynamiques possède(nt) une ou des Foreign Keys
[extra] explicite la relation entre
FK: nom d'une cle FK dans [dynamique]
FKP: le nom de la PK vers laquelle pointe FK
tables: la table de reference pour FKP
si plusieurs, l'odre horizontal doit etre respecté table[0], FK[0], FKP[0] etc
permet de détecter toutes les FK d'une table dynamique 

[extra] 
tables=spip_vap_antenne,spip_vap_regions 
FK=lienant,lienreg 
FKP=idant,idreg

statique:

Si relation sql: multi2multi : l'autre table source de donnees necessaires au contexte.

tables, nom de la table statique
PK : clé primaire de la table statique
champs : nom de colonne contenant la valeur statique
valeurs : les valeurs exclusives provenant du formulaire html et attachées à [statique][champs]
La valeur (choix exclusif) issue de ce formulaire sera codée dans une des colonnes sql indiquée dans [liaison][champs]

[statique]
tables=spip_vap_transpu
PK=idtrans
champs=societes
valeurs=oui,ouiavec

liaison:

Si relation sql multi2multi, [liaison] explicite la liaison entre la table dynamique principale et la table statique.

table : le nom de la table de liaison
PK: sa Primary Key
FK : les clés Foreign pointant vers les tables (dynamique et statique), le respect de l'ordre vertical est important: FK[0]==FKP[0], etc
champs : colonnes spécifique à la table de liaison. Le choix d'une colonne est exclusif.
valeurs: selon le choix de [statique][valeurs], indique la valeur positive à placer dans [liaison][champs] 

[liaison]
table=spip_vap_mobiles
PK=idmob
FK=lienhum,lientrans
FKP=idhum,idtrans
champs=utilisation,abonne
valeurs=oui

passive :

Si relation multi2one: les valeurs [passive][champs] sont selectionnees depuis [passive][tables] via [passive][PK] et fournies en supplement au contexte
permet la creation d'un cache de donnee 'passives' à servir à chaque fois dans ce contexte
pas de FK ici
pour une meme table, je ne gère que un fichier passif...
du à un cache simpliste, le système ne sait gérer qu'un seul et unique ajout 'passif' pour tous les contextes 

[passive]
tables=spip_vap_antenne
PK=idant  
champs=idant,nomantenne,lienreg
